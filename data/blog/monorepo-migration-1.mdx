[FEConf 2025](/blog/feconf-2025) 네트워킹 주제로 모노레포가 자주 나왔다. 모노레포는 여러 프로젝트에서 공통으로 쓰는 패키지, 함수, UI를 한 곳에서 관리하는 방식이라고 한다.

이야기를 들으면서 회사 프로젝트가 딱 떠올랐다. consumer / provider / operator 세 개의 독립 레포가 있는데, 3곳이 거의 동일한 컴포넌트와 설정을 각자 유지하고 있었다. 코드 중복률 90% 이상이었다.

> 이 글은 4편 시리즈 중 1편입니다. 전환을 결정하기까지의 조사와 계획 과정을 다룹니다.

---

## 왜 모노레포인가

consumer, provider, operator.
3개의 앱은 사용 타겟층은 다르지만 같은 디자인 시스템, 컴포넌트, API 호출 로직을 거의 동일하게 공유하고 있었다.

`Button.vue`, `Empty.vue`, `useToast.ts` 같은 파일들이 세 레포에 각각 존재했다. 완전히 동일하지도 않았다.
A 레포에서 수정한 걸 B, C에 반영하는 걸 놓치면 같은 이름의 파일인데 내용이 달라지기 시작했다.

패키지 버전도 마찬가지였다. Nuxt만 해도 consumer는 3.11.2, provider는 3.14.1이었고 Node.js도 어떤 레포는 20, 어떤 레포는 24를 쓰고 있었다. 같은 회사, 같은 제품군인데 실행 환경이 제각각이었다.

공통 컴포넌트만 각 레포 15개씩, <b>총 파일이 45개, 코드 중복률이 90% 이상이었다.</b>
공통 컴포넌트 하나를 수정하려면 세 레포에서 각각 작업하고 PR을 올리고 머지해야 했다.
레포마다 분명 같은 컴포넌트로 세팅했지만 각각 수정사항들이 들어오면서 어떤 레포 수정을 놓쳤는지, 작은 차이들이 점점 많아지면서 관리하기가 어려워지고 있다는 생각이 들었다.

### 기존 구조 (3개 독립 레포)

```
consumer-repo/               provider-repo/               operator-repo/
├── components/              ├── components/              ├── components/
│   ├── Button.vue           │   ├── Button.vue           │   ├── Button.vue
│   ├── Empty.vue            │   ├── Empty.vue            │   ├── Empty.vue
│   ├── Modal.vue            │   ├── Modal.vue            │   ├── Modal.vue
│   └── ...                  │   └── ...                  │   └── ...
├── composables/             ├── composables/             ├── composables/
│   ├── useToast.ts          │   ├── useToast.ts          │   ├── useToast.ts
│   └── useAuth.ts           │   └── useAuth.ts           │   └── useAuth.ts
├── assets/                  ├── assets/                  ├── assets/
├── pages/                   ├── pages/                   ├── pages/
├── nuxt.config.ts           ├── nuxt.config.ts           ├── nuxt.config.ts
├── tsconfig.json            ├── tsconfig.json            ├── tsconfig.json
├── eslint.config.js         ├── eslint.config.js         ├── eslint.config.js
└── package.json             └── package.json             └── package.json
```

세 레포가 거의 같은 파일을 각자 갖고 있었다. `Button.vue`를 수정하면 세 레포에 각각 PR을 올려야 했다.

## 기술 스택 선택

기존 npm 에서 **pnpm workspace + Turborepo** 조합으로 결정했다.

npm도 workspace를 지원하지만 **pnpm**은 패키지를 글로벌 저장소에 한 번만 저장하고 심볼릭 링크로 참조하는 방식이라 설치 속도가 2-3배 빠르다.
세 앱의 의존성이 많이 겹치는 상황이라 이 차이가 체감될 것 같았다.

빌드 시스템은 <b>Turborepo</b>로 정했다. 변경이 없는 패키지는 캐시를 쓰고, 의존성이 없는 작업은 병렬로 돌린다.
세 앱을 한 레포에서 빌드하면 시간이 늘어날 수 있는데, Turborepo가 수정이 있는 패키지만 빌드하도록 세팅해 이 문제를 잡아줄 거라 봤다.

```json
// turbo.json — packages/shared가 먼저 빌드되고, 그 다음에 각 앱이 빌드됨
{
  "$schema": "https://turbo.build/schema.json",
  "tasks": {
    "build": {
      "dependsOn": ["^build"],
      "outputs": [".output/**"]
    },
    "dev": {
      "cache": false,
      "persistent": true
    }
  }
}
```

## 위험 요소

계획을 세우면서, 잘못될 가능성이 있는 것들을 먼저 적어봤다.

모노레포로 전환하려는 consumer, provider, operator 세 앱 모두 고객이 쓰고 있는 상태라,
전환 작업이 진행되는 동안에도 세 앱이 계속 정상적으로 배포되고 운영돼야 했다. 개발 서버가 별도로 운영되고 있지만,
중간중간 들어오는 수정사항도 계속 반영해야 했다. **모노레포 전환 작업이 배포나 운영에 영향을 주면 안 됐다.**

**각각 독립 배포가 가능해야 한다.** 각 앱이 서로 다른 고객을 대상으로 하고, 수정 사항이 각자 다르게 들어오는 상황이라, 세 앱이 한 레포에 있더라도 독립적으로 빌드되고 배포될 수 있도록 세팅해야 했다.

3개 레포에서 공통으로 사용하는 `packages/shared` 변경의 파급 범위도 문제였다. 공통 컴포넌트 하나를 수정하면 세 앱 전체에 영향이 간다.
모노레포가 아닐 때는 각 레포가 분리돼 있어서 영향 범위가 제한적이었지만, **통합되면 shared 파일 하나에 오류가 생기면 세 앱을 동시에 깨뜨릴 수 있다.**

빌드 시간 증가도 변수였다. **파일 크기가 커지면서 한 레포에서 돌리면 전체 빌드 시간이 길어질 수 있다.**
Turborepo 캐싱으로 어느 정도 커버가 될 거라 봤지만, 일단 해봐야 알 것 같았다.

## 11주 계획

기존 레포에서 작업하는 게 아닌 **로컬에서 새로 모노레포를 만들어서 완성된 후에 한 번에 옮기는 방식**으로 계획을 세웠다.
최종적으로 모노레포를 배포했을 때, 고객은 아무런 차이를 느끼지 못해야 했다.

한번에 모든 것을 작업하기엔 위험이 크다고 판단해서, 단계별로 나눠서 진행하고, 단계마다 배포해서 검증하는 방식으로 계획을 세웠다.
전환 작업을 6단계로 나눴다.

| 단계    | 내용                                 | 예상 기간 |
| ------- | ------------------------------------ | --------- |
| 1단계   | 기본 구조 생성, 레포 이동            | 1주       |
| 2-3단계 | 공통 패키지 추출, @shared 설정       | 4주       |
| 4단계   | 설정 파일 통합 (tsconfig, ESLint 등) | 2주       |
| 5단계   | CI/CD 파이프라인 재구성              | 2주       |
| 6단계   | E2E 테스트 구축, 마무리              | 2주       |

총 11주. 성공 기준도 미리 정해뒀다. **중복 컴포넌트 단일화, 세 앱 독립 배포 유지, 빌드 시간 기존 대비 유지.** 이 셋이 충족되면 성공이라고 보기로 했다.

예상 효과도 계산해봤다. 공통 컴포넌트 수정 한 번에 3-4시간 걸리던 게, 모노레포에서는 30분-1시간으로 줄어들 것으로 예상했다. 실제로 그렇게 됐는지는 4편에서 확인할 수 있다.

계획은 세웠다. 이제 진짜 시작이었다.

결과부터 말하자면, 11주 계획은 실제로 3개월이 걸렸다.그리고 예상하지 못했던 문제들이 중간에 꽤 있었다.
