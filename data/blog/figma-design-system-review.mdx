디자이너가 피그마에서 색상 하나 바꿨을 때, 개발자가 일일이 CSS를 수정하지 않아도 되는 환경을 만들고 싶었습니다.

**Tokens Studio** + **Style Dictionary** + **TailwindCSS v4** + **GitHub Actions**로 피그마 토큰이 코드에 자동으로 반영되는 워크플로우를 실제로 구축했고, 충분히 동작하는 것도 확인했습니다. 그런데 결론은 도입하지 않기로 했습니다.

이 글은 왜 그런 결정을 했는지, 그리고 구축 과정에서 무엇을 배웠는지에 관한 기록입니다.

---

## 🔄 구축한 워크플로우 아키텍처

전체 흐름은 다음과 같습니다.

> 피그마 Tokens Studio → GitHub Repository → GitHub Actions → Style Dictionary → TailwindCSS v4 → 웹사이트

단계별로 설명하면:

1. **디자인 토큰 정의 (피그마)**: Tokens Studio 플러그인으로 컬러 팔레트, 스페이싱, 타이포그래피, Border Radius 등을 정의
2. **토큰 동기화 (GitHub)**: 피그마에서 "Push to GitHub" 클릭 → `tokens/global.json`에 자동 저장
3. **자동 빌드 (GitHub Actions)**: 토큰 변경 감지 → Style Dictionary로 CSS/TypeScript 파일 생성
4. **코드 통합 (TailwindCSS v4)**: 생성된 테마 파일이 프로젝트에 자동 적용

## 🛠️ 기술 스택

**Frontend**

- **Nuxt.js 3.8.0** — Universal Vue.js Framework
- **Vue.js 3** — Progressive JavaScript Framework
- **TypeScript 5.3.3** — 타입 안전성 보장

**디자인 토큰 관리**

- **Tokens Studio for Figma** — 피그마 플러그인으로 토큰 정의
- **Style Dictionary 4.0.1** — 토큰 변환 라이브러리
- **TailwindCSS v4.1.6** — CSS 프레임워크

**자동화 도구**

- **GitHub Actions** — CI/CD 파이프라인
- **GitHub Pages** — 토큰 쇼케이스 정적 배포
- **Nodemon** — 개발 중 파일 변경 감지

## 📁 프로젝트 구조

```
figma_mcp/
├── tokens/
│   └── global.json                 # 피그마에서 생성된 토큰 원본
├── assets/css/
│   ├── design-tokens.css          # 생성된 CSS 변수
│   ├── theme-tokens.css           # TailwindCSS v4 테마
│   └── main.css                   # 메인 스타일시트
├── components/
│   └── tokens/                    # 토큰 표시용 컴포넌트들
├── composables/
│   ├── useDesignTokens.ts         # 생성된 TS 타입
│   ├── useTokens.js               # 토큰 관리 로직
│   └── useTokenExtractors.js      # 토큰 추출 유틸리티
├── pages/
│   └── index.vue                  # 토큰 쇼케이스 페이지
├── .github/workflows/
│   ├── build-and-deploy.yml       # 메인 배포 워크플로우
│   └── validate-tokens.yml        # 토큰 검증 워크플로우
├── transform-tokens.js            # 토큰 변환 스크립트
└── nuxt.config.ts                 # Nuxt 설정 (SPA 모드)
```

## 🚀 사용 방법

### 1. 피그마에서 토큰 수정

1. 피그마에서 **Tokens Studio 플러그인** 실행
2. 토큰 값 수정 (색상, 간격, 폰트 등)
3. **"Push to GitHub"** 버튼 클릭

### 2. 자동 프로세스 실행

버튼 한 번으로 아래가 자동으로 돌아갑니다.

1. GitHub Actions가 자동으로 트리거됨
2. `transform-tokens.js`가 실행되어 Style Dictionary가 토큰을 CSS/TS 파일로 변환
3. 변경사항이 자동으로 커밋됨
4. GitHub Pages에 새 버전 배포

### 3. 코드에서 토큰 사용

**CSS 변수 방식:**

```css
.my-component {
  color: var(--colors-spring-500);
  padding: var(--spacing-md);
  border-radius: var(--border-radius-box-md);
}
```

**TailwindCSS 클래스 방식:**

```html
<div class="bg-spring-500 p-md rounded-box-md">
  <!-- TailwindCSS v4에서 자동 생성된 클래스 -->
</div>
```

### GitHub Actions 워크플로우

**Build & Deploy** (`figma_style` 브랜치 푸시 시 트리거)

1. 토큰 파일 변경 감지
2. `transform-tokens.js` 실행
3. CSS/TS 파일 생성
4. 변경사항 자동 커밋
5. GitHub Pages 배포

**Token Validation** (Pull Request 생성 시 트리거)

1. `tokens/global.json` 구문 검증
2. 필수 토큰 속성 확인
3. 토큰 구조 유효성 검사

## 🎯 기대했던 도입 효과

**디자이너 관점**

- 피그마에서 직접 토큰 관리 가능
- 변경사항이 즉시 개발 환경에 반영
- 일관성 있는 디자인 시스템 유지

**개발자 관점**

- 수동 CSS 업데이트 불필요
- TypeScript 타입 안전성 보장
- TailwindCSS와 완벽 통합

**팀 협업 관점**

- 디자인-개발 간 싱크 자동화
- 토큰 변경 이력 Git으로 추적
- 실시간 토큰 쇼케이스 페이지 제공

이 모든 것이 실제로 동작하는 것을 확인했습니다. 그런데도 도입하지 않기로 했습니다.

---

## ❌ 도입하지 않기로 한 4가지 이유

### 1. 비용 대비 효과 부족

실무에서 가장 필요한 <b>컴포넌트 단위 관리</b>는 Tokens Studio **PRO 버전(월 49유로)** 에서만 지원됩니다. 무료 버전과 PRO 버전의 핵심 차이가 오직 컴포넌트 지원 여부뿐인데, 현재 테스트 단계에서 이 비용은 과도하다고 판단했습니다.

### 2. 무료 버전의 한계

무료 버전으로는:

- Token 기반 관리(색상, 간격, 폰트 등)만 가능
- **그룹 단위 스타일 관리 불가능** — 실무에서 필요한 핵심 기능 부재
- 디자인팀이 요구하는 수준의 관리 체계 구현 어려움

### 3. 업무 효율성 저하 우려

무료 버전으로 제대로 쓰려면 <b>추가 JSON 파일 작성</b>이 필요합니다. 이는 기존 작업 방식보다 **오히려 업무가 증가**할 가능성이 있었고, "편의성 향상"이라는 도입 목적과 정반대였습니다.

### 4. 대안 솔루션 존재

컴포넌트 관리는 **Storybook** 등 이미 검증된 도구로 충분히 커버할 수 있었고, 현재 개발 단계에서는 기존 방식으로도 충분히 관리 가능했습니다.

---

## 🔮 향후 계획

지금 당장 도입하지 않는다는 결정이지, 이 워크플로우를 버린다는 뜻은 아닙니다.

- **확장성 관점**: 다른 프로젝트나 팀 규모가 확대될 경우 재검토할 예정입니다.
- **비용 효율성**: PRO 버전의 가격 정책이 변동되거나 팀 규모가 성장하면 재평가할 계획입니다.
- **기술 검증**: 구축한 워크플로우는 향후 도입 시 바로 활용 가능한 자산으로 보관해 둡니다.

---

## 💡 얻은 인사이트

이번 검토를 통해 배운 것들입니다.

- **기술적 가능성과 비즈니스 타당성은 별개다** — 잘 동작하는 것과 도입할 만한 것은 다름
- **무료 도구의 한계를 명확히 파악하고 실무 요구사항과 비교 검토해야 한다** — 체험 전에 PRO 기능 목록을 꼭 확인할 것
- **도입 목적과 실제 결과 간의 괴리를 사전에 따져봐야 한다** — "편리하게 하려고 도입했더니 할 일이 늘었다"는 최악의 시나리오
