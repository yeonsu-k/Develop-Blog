블로그 글을 작성합니다. 아래 규칙을 따르되, 글 유형에 맞는 구조를 선택하세요.

---

## 절대 하지 말아야 할 것

### 구조 관련

- 섹션 제목에 이모지 사용 금지 (💡✅🛠️⚙️💻 등)
- 모든 글에 동일한 템플릿 적용 금지 (설정→기능→예시→후기→참고자료 구조 반복 금지)
- 수평선(`---`)만으로 섹션을 끊는 것 금지
- 하나의 생각을 억지로 불릿 포인트로 나열하는 것 금지

### 문체 관련

- "~할 수 있습니다", "~하는 방법을 소개합니다" 같은 설명서 투 문체 금지
- "먼저", "다음으로", "마지막으로" 순서 나열 금지
- "강력히 추천합니다", "매우 유용합니다" 같은 과장된 평가 금지
- 결론 섹션에서 본문을 그대로 요약하는 것 금지

## 반드시 지켜야 할 것

### 개인 서사를 넣어라

- 왜 이것을 써봤는지 동기를 자연스럽게 설명한다
- 처음에 기대했던 것과 실제로 겪은 것의 차이를 구체적으로 쓴다
- 삽질했거나 헷갈렸던 부분이 있으면 솔직하게 언급한다
- "나는 이게 이래서 좋았다/별로였다"는 주관적 판단을 명확히 드러낸다

### 문체

- 줄글 위주로 쓴다. 불릿은 정말 나열이 필요할 때만 사용한다
- 섹션 간 이동할 때 자연스러운 연결 문장을 넣는다 ("그래서 직접 해봤다", "기대보다 훨씬 달랐던 건 이 부분이었다" 등)
- 완벽하게 정제된 문장보다 생각의 흐름이 느껴지는 문장이 낫다
- 짧고 단호한 문장과 설명하는 문장을 섞는다
- 한 문장이 3줄 이상 이어지지 않는다. 길면 끊는다

### 구조

- 글마다 다른 흐름을 갖는다. 후기 글은 후기처럼, 튜토리얼은 튜토리얼처럼 구조가 달라야 한다
- 설정 방법을 포함할 경우, 설정 중에 마주친 문제나 주의사항을 경험 기반으로 섞어서 설명한다
- "참고 자료" 섹션을 기계적으로 붙이지 않는다. 언급이 필요한 링크는 본문 흐름 안에 자연스럽게 녹인다

## 글의 성격에 따라 톤을 구분한다

- **튜토리얼 / 설정 가이드**: 경어체(~합니다, ~하세요) 유지. 독자에게 설명하는 느낌
- **후기 / 인상 / 개인 의견**: 구어체(~했다, ~이다, ~더라) 사용. 친구한테 말하듯이 쓴다
- 하나의 글 안에서 섞는 것도 가능하다. 단, 같은 섹션 안에서는 통일한다

**캐주얼 톤 예시 (후기 / 인상 섹션):**

> 솔직히 처음엔 별 기대 안 했다. 어차피 비슷한 거 아닌가 싶었는데, 막상 써보니 생각보다 훨씬 편했다.
> 특히 빌드 오류를 알아서 잡아주는 부분은 좀 놀라웠다. "이게 되네?" 하는 느낌.

**피해야 할 패턴 (캐주얼 톤인데 AI 같은 경우):**

> 이 기능은 매우 유용하며, 개발 생산성을 크게 향상시킬 수 있습니다. 적극 활용을 권장합니다.

## 공통 형식 규칙

**강조:**

- `<b>` 태그 또는 `**텍스트**` 마크다운으로 핵심 키워드·강조 표현 (문단당 1~2개 이하)
- `>` 인용구는 팁·주의사항에 사용

**코드 블록:**

- 언어 태그 필수 (json, bash, css 등)

**이미지 삽입 형식:**

```mdx
<div className="flex flex-col items-center justify-center gap-3 md:flex-row md:gap-5">
  <img
    src="/static/images/blog/{폴더}/{파일}.png"
    alt="{설명}"
    className="!mt-0 !mb-0 w-full rounded-lg shadow-sm sm:w-11/12 md:w-[50%]"
  />
</div>
```

이미지 1장이면 `md:w-[50%]` 제거, 이미지 없으면 블록 생략

**Frontmatter:**

```yaml
---
title: '{제목}'
date: '{날짜}'
tags: ['{태그1}', '{태그2}']
draft: false
summary: '{1줄 요약}'
images: []
authors: ['default']
---
```

## 글 유형별 구조

### A. 설정/가이드형 (MCP 시리즈, 도구 설치·설정)

```
{인트로: 이 도구가 무엇이고 왜 연결/설치했는지. 2~3문장 줄글}

## 설정 방법

### 1. {사전 준비}
### 2. {설치}
### 3. {설정 파일 작성}

> 💡 **중요:** {주의사항 — 실제 겪은 것 기반}

## 주요 기능

{기능 설명은 불릿 나열 대신, 어떤 상황에서 어떻게 썼는지 흐름으로 설명}

## 직접 써보니

{실제 사용 경험. 구어체 가능. 기대와 달랐던 점, 유용했던 순간 구체적으로}
{이미지}
```

---

### B. 경험/후기형 (사용 후기, 컨퍼런스, 프로젝트 회고)

```
{인트로: 배경/동기/맥락 2~3문장. 결말을 미리 암시해도 됨. 구어체 가능}

## {주제에 맞는 섹션 — 정해진 틀 없음}

{섹션마다 --- 구분선은 흐름상 필요할 때만}
```

- **컨퍼런스 후기**: 세션별 소제목(`### 세션명`)으로 구성, 개인 감상 자연스럽게 섞음
- **프로젝트 회고**: 문제 → 과정 → 결과 → 인사이트 흐름으로 구성

## 글의 완성 기준

다 쓴 후 스스로 다음을 확인한다:

1. 이 글을 쓴 사람이 직접 써봤다는 게 느껴지는가?
2. 섹션 구조가 다른 글과 비슷하지 않은가?
3. 불릿 포인트를 줄글로 바꿔도 자연스러운가? (자연스럽다면 줄글로 바꿔라)
4. 결론이 본문의 요약이 아닌, 쓰고 나서 남은 생각인가?

**가독성**

5. 문단이 너무 길지 않은가? 한 문단은 3~5문장을 넘지 않도록 한다
6. 코드 블록, 인용구, 이미지 등 시각적 요소가 텍스트 사이에 적절히 분배되어 있는가?
7. 핵심 단어나 문장은 `<b>` 태그 또는 `**텍스트**`로 강조되어 있는가? 남용하지 않는다 (문단당 1~2개 이하)
8. 제목만 훑어봤을 때 글의 흐름이 파악되는가?

**블로그 문체**

9. 경어체와 구어체 중 하나로 섹션 내 통일되어 있는가?
10. 전문 용어를 처음 등장할 때 짧게 설명하거나 독자가 알 것으로 전제하는지 기준이 일관적인가?
11. 한 문장이 3줄 이상 이어지지 않는가? 길면 끊는다
12. 글을 소리 내어 읽었을 때 어색한 부분이 없는가? (번역투, 문어체 감지)
13. 후기나 인상을 서술하는 섹션은 구어체로 작성되었는가?
14. 캐주얼 톤 섹션에서 "~할 수 있습니다", "~권장합니다" 같은 격식체가 섞이지 않았는가?

## 데이터 소스 안내

- **노션 기반 글**: 노션 MCP(`notion_retrieve_block_children`)로 해당 페이지 ID의 블록을 읽어 내용 파악
- **로컬 파일 기반 글**: 지정된 파일 경로를 직접 읽어 내용 파악
- **두 소스 합치는 경우**: 노션 내용을 뼈대로, 로컬 파일로 디테일 보완

작성할 글의 유형(A/B)과 데이터 소스(노션 페이지 ID 또는 파일 경로)를 알려주세요.
